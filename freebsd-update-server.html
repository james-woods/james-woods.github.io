Build your own FreeBSD Update server

Fr 19 Dezember 2014

FreeBSD is an operating system that we use to various things at work. It's most significant advantage lays in the adavanced filesystem ZFS and it's proven stability. As of this writing we run 39 servers with FreeBSD and we update them regularly. While normal patches are quite fast to fetch and install, the upgrade from version 9.2 to 9.3 took a significant amount of time to fetch all the necessary files from the official servers.

That is why we started to run our own internal FreeBSD update server. There exist some documentation for this inside the official handbook which formed the basis for this blog post.

Prerequisites

Important to know is that the update server has to run a newer version than the version you are trying to distribute. Thus I first upgraded the update server machine to 9.3 via the freebsd-update -r 9.3-RELEASE upgrade command as described in point 24.2.3 in the handbook.

When running on the newest version, proceed with checking out the SVN repository that holds all the configuration and scripts for the update server via:

cd /usr/local/
svn co http://svn.freebsd.org/base/user/cperciva/freebsd-update-build freebsd-update-server

I suggest to use /usr/local as this is the 'standard' path of most scripts within the repository.

Building a version

As time of this writing there exists no default config in the /usr/local/freebsd-update-server/scripts folder. That is why we copy over the configuration from 9.2 first:

cp -r /usr/local/freebsd-update-server/scripts/9.2-RELEASE /usr/local/freebsd-update-server/scripts/9.3-RELEASE

Next, as we want to build and distribute an amd64 version, we create a folder for that, copy over the configuration from i386 and remove the i386 folder:

cd /usr/local/freebsd-update-server/scripts/9.3-RELEASE
mkdir amd64
cp i386/build.conf amd64/build.conf
rm -rf i386

The last step is of course optional. I like to keep things tidy and thus removed the i386 folder. Next we edit the amd64/build.conf file and adjust the SHA256 hash and the EOL date in it to reflect version 9.3:

# SHA256 hash of disc1.iso image.
export RELH=5a3c82653d77bba7d7ded8bd7efbedc09d52cf4045d98ce52a82c9e0f8fa9b0e

# Components of the world, source, and kernels
export WORLDPARTS="base doc games"
export SOURCEPARTS="src"
export KERNELPARTS="kernel"

# EOL date
export EOL=1483142400

FreeBSD upgrades are signed by the update server. This is done via a key that is generated via the first run of the make script:

cd /usr/local/freebsd-update-server
sh scripts/make.sh

Note that password down somewhere safe, you will need it to sign all updates in the future. Next step is to run our initial build. This can take some time thus I recommend starting a tmux (or screen if you like) session first:

tmux new -s upgrade_server
cd /usr/local/freebsd-update-server

We will need to adjust the scripts/build.conf so that the script can fetch the correct ISO image for our version 9.3, adjust the following line like this:

# Location from which to fetch releases
export FTP=ftp://ftp.freebsd.org/pub/FreeBSD/releases/ISO-IMAGES/9.3/

After that we can start our first build:

sh scripts/init.sh amd64 9.3-RELEASE

This process can take up to several hours, depending on the hardware it is run on. It finishes with the following output:

FreeBSD/amd64 9.3-RELEASE initialization build complete.  Please
review the list of build stamps printed above to confirm that
they look sensible, then run
# sh -e approve.sh amd64 9.3-RELEASE
to sign the release.

First, we mount our signing key with the password that we encrypted it with, then we sign our release:

sh -e scripts/mountkey.sh
sh -e scripts/approve.sh amd64 9.3-RELEASE

This finishes with:

The FreeBSD/amd64 9.3-RELEASE update build has been signed and is
ready to be uploaded.  Remember to run
# sh -e umountkey.sh
to unmount the decrypted key once you have finished signing all
the new builds.

Next we continue with the 9.2 update files, which have to be built as well. This is necessary since the system has to determine the difference between two versions. For this we have to create the amd64 directory in the scripts folder and copy over the build.conf from the i386 folder like we already did for the 9.3 files:

mkdir scripts/9.2-RELEASE/amd64
cp scripts/9.2-RELEASE/i386/build.conf scripts/9.2-RELEASE/amd64/build.conf

Next we edit the amd64/build.conf file for 9.2 and adjust the SHA256 hash in it :

# SHA256 hash of disc1.iso image.
export RELH=a8c1751b83646530148766618a89a97009e7500e7057a5cbe3afd74ef480c915

We will need to adjust the scripts/build.conf so that the script can fetch the correct ISO image for the version 9.2, adjust the following line like this:

# Location from which to fetch releases
export FTP=ftp://ftp.freebsd.org/pub/FreeBSD/releases/ISO-IMAGES/9.2/

And fire off our build of the 9.2-RELEASE:

sh scripts/init.sh amd64 9.2-RELEASE

This will complete with:

FreeBSD/amd64 9.2-RELEASE initialization build complete.  Please
review the list of build stamps printed above to confirm that
they look sensible, then run
# sh -e approve.sh amd64 9.2-RELEASE

To mount the key and sign the release run:

sh -e scripts/approve.sh amd64 9.2-RELEASE

Then build the upgrade patches between 9.2 and 9.3 via:

sh tools/build-upgrade-patches.sh pub/ amd64 9.3-RELEASE 9.2-RELEASE

This command should finish without any output. We now have all necessary files to upgrade a 9.2 system to a 9.3 system. To distribute these we will use the webserver nginx.

Installing and configuring nginx

Installing nginx via the pkg package manager starts with:

pkg install nginx

To enable it add

nginx_enable="YES" to

to the \etc\rc.conf file.

service nginx start
